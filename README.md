## 🎤 Interview Corner (Ответы на вопросы из ТЗ)

### 1. Почему в JSX используется className, а не class?
Поскольку React работает на JavaScript, слово `class` является зарезервированным ключевым словом для создания классов объектов. Чтобы браузер не путал CSS-классы с логикой языка, в JSX используется атрибут `className`.

### 2. Чем props отличаются от state?
*   **Props** — это внешние данные, которые компонент получает от родителя (как "Техническое задание"). Они неизменны для самого компонента.
*   **State** — это внутренняя память компонента (его "текущее состояние"). Изменение state инициализирует перерисовку интерфейса.

### 3. Почему props нельзя изменять внутри компонента?
В React действует принцип однонаправленного потока данных. Если бы компоненты могли сами менять входящие props, это привело бы к непредсказуемым ошибкам в работе всей системы и нарушило бы "цифровую нить" данных.

### 4. Почему state нужно изменять через setter-функцию, а не прямым присваиванием?
Прямое присваивание (например, `state = 5`) — это обычная смена значения переменной в памяти, о которой React "не узнает". Setter-функция (например, `setCount`) уведомляет React об изменениях, что заставляет его обновить Virtual DOM и отобразить изменения на экране.

### 5. Что такое управляемый компонент и почему он важен для форм?
Это компонент (например, наш поиск), значение которого (`value`) полностью синхронизировано со `state`. Это критически важно, так как позволяет React мгновенно реагировать на ввод (например, фильтровать список заявок "на лету") и всегда иметь актуальные данные для отправки.

### 6. Как Вы понимаете формулу UI = f(state)?
Это фундаментальный принцип React: интерфейс (UI) является результатом (функцией) текущего состояния данных (state). Мы не меняем кнопки или текст вручную — мы меняем данные в стейте, а интерфейс подстраивается под них автоматически.

### 7. Зачем компонентам в React нужен key при рендеринге списка через map()?
`key` — это уникальный идентификатор элемента. Он позволяет React точечно обновлять, перемещать или удалять конкретную карточку задачи в DOM, не перерисовывая весь список целиком. Это обеспечивает высокую производительность приложения.

## 🏗 Декомпозиция проекта
1.  **App (Root)** — Хранит массив заявок и текущую роль пользователя.
2.  **SimulationRequestCard** — Карточка объекта испытаний (LOD-индекс, параметры модели).

4.  **SearchBar** — Управляемый фильтр для поиска по ID или названию.
